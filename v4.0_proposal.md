## 1. Game Overview

**Neo-Hack: Gridlock v4.0** elevates the turn-based asymmetric cyber warfare strategy game by introducing **Synchronous Multiplayer** (1v1, 2v2, Co-op) and **Enhanced Autonomous Agents**. The core conflict between Attacker and Defender expands from single-player vs AI into a dynamic ecosystem where human players can battle each other, team up against advanced neural-network-driven AI, or deploy custom AI Sentinels to wage proxy wars.

- **Genre:** Turn-based multiplayer strategy, cybersecurity simulation
- **Theme:** Near-future dystopian hacking / cyber warfare
- **Visual Style:** Dark terminal/CRT aesthetic with neon accents (cyan, red, amber)
- **Platform:** Web browser (desktop-first with mobile companion), Vite + React/Svelte frontend, FastAPI + WebSockets + SQLModel backend

## 2. Core Game Rules & Multiplayer Integration

### 2.1 Turn Structure (Synchronous & Asynchronous)

To accommodate multiplayer without slowing down gameplay, v4.0 introduces a **Simultaneous Resolution** model for certain phases, alongside traditional turn-based play.

| Property | Value |
|---|---|
| Turn order | **Planning Phase (Simultaneous):** Both sides lock in actions.<br>**Resolution Phase:** Actions execute sequentially based on a new Initiative stat. |
| Action Points (AP) | Scales dynamically. Default 2 AP. Deploying Sentinels costs max AP. |
| Turn increment | After **Resolution Phase** completes, `current_turn++` |
| Time Limits | Multiplayer enforces a strict 60-second Planning Phase timer. |

### 2.2 Network Topology & Dynamics

- Networks are procedurally generated or selected from balanced competitive templates.
- **Dynamic Topology:** Nodes can now be dynamically re-routed. Defenders can temporarily sever physical connections (`ISOLATE_LINK`), altering the adjacency matrix mid-game.

### 2.3 Fog of War Updates

- **Attacker:** Maintains limited visibility. Can now deploy `HONEYPOTS` (fake signals) to confuse defenders.
- **Defender:** Visibility is no longer absolute. Attackers can deploy `ROOTKITS` to hide nodes from the Defender's map until a `DEEP_SCAN` is performed.

## 4. Stealth, Detection, and Deception

### 4.1 The Paranoia System

Replacing the linear Alert Level from v3.2.0, v4.0 introduces the **Paranoia Meter**.
- **Location-based Alertness:** Alert level is no longer strictly global. Actions generate "heat" on specific subnets.
- Defenders receive visual "Heatmaps" rather than exact node alerts when detection occurs, requiring them to triangulate the attacker or rely on Hunter AIs.

---

## 5. Resource Economy

### 5.1 Compute Cycles (New Global Resource)

Both factions now passively generate **Compute Cycles (CC)** each turn based on the number of nodes they control.
- CC is used to purchase advanced actions, deploy Sentinels, and bid on neutral black-market tools during the Planning Phase.
- **Economy Warfare:** Players can attack "Compute Nodes" specifically to cripple the opponent's resource generation.

## 9. Sentinel Nexus (Enhanced AI Agents)

The Sentinel Lab from v3.2.0 is evolved into the **Sentinel Nexus**, shifting from basic parameter sliders to a modular behavior tree and machine-learning deployment system.

### 9.1 Sentinel Architectures

Players construct AI agents using "Logic Blocks" unlocked via gameplay.
- **Triggers:** "If node detected", "If Paranoia > 50", "If CC > 100"
- **Behaviors:** "Prioritize Stealth", "Swarm Target", "Turtle Defense"

### 9.2 Machine Learning (PPO) Integration

- **Training Gym:** Players can submit their Sentinel architectures to the cloud. The backend (utilizing Ray RLlib) runs rapid simulated games against baseline AIs, returning a slightly mutated, highly optimized "Evolved Sentinel" after 24 hours.
- **Marketplace:** Players can trade or sell highly successful Evolved Sentinels to other players using in-game currency.

## 14. WebSocket Event Architecture

Replacing the REST polling from v3.2.0, v4.0 utilizes an authoritative WebSocket server for real-time multiplayer synchronization.

### 14.1 WebSocket Payloads (JSON-RPC 2.0 style)

**Client → Server (Action Submission):**
```json
{
  "type": "ACTION_LOCK",
  "payload": {
    "action_id": 1,
    "target_node": 14
  }
}
```

**Server → Client (Resolution Broadcast):**
```json
{
  "type": "TURN_RESOLUTION",
  "payload": {
    "turn_number": 5,
    "actions": [
      { "player": "attacker_1", "action": "EXPLOIT", "target": 14, "success": true },
      { "player": "defender_1", "action": "PATCH", "target": 14, "success": false }
    ],
    "new_state": { ... }
  }
}
```

## 16. Design System & UI Reengineering

Addressing Section 20 of the v3.2.0 spec, v4.0 enforces modern frontend architecture.

### 16.1 Component Framework
- Fully migrate from Vanilla JS/Template Strings to **React 18** or **Svelte 4**.
- Use **Zustand** (React) or built-in stores (Svelte) for the global `GameState`.

### 16.2 Responsive Design
- The UI is redesigned with CSS Grid/Flexbox to support tablet views.
- **Mobile Companion Mode:** A dedicated mobile view allowing players to tweak Sentinels or review replays on the go.
